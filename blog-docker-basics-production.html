Docker Basics That Actually Matter in Production

By Imran Ali ‚Ä¢
Reality check: Docker commands are easy. Running Docker reliably in production is not. This guide focuses only on what truly matters in real environments.

1) Images: small, predictable, reproducible

  • Always pin versions (avoid :latest)
  • Use minimal base images (alpine/distroless when possible)
  • One process per container
FROM nginx:1.25-alpine
COPY nginx.conf /etc/nginx/nginx.conf

2) Containers are disposable

If restarting a container causes data loss, your design is wrong. Containers should be replaceable at any time.

Golden rule: Never store important data inside the container filesystem.

3) Volumes: data lives outside containers

docker run -d \
  -v /data/mysql:/var/lib/mysql \
  mysql:8.0

4) Networking: explicit is better

  • Avoid using the default bridge blindly
  • Create user-defined networks
  • Expose only required ports
docker network create app-net
docker run -d --network app-net --name web nginx

5) Logging: stdout is king

Applications should log to stdout/stderr. Docker handles log collection; external tools handle storage. 

docker logs web
docker logs -f web

6) Resource limits prevent outages

docker run -d \
  --memory="512m" \
  --cpus="1.0" \
  myapp:1.0
Production failure pattern: One noisy container consumes all RAM and kills the host.

7) Basic security that people ignore

  • Never run containers as root if avoidable
  • Scan images for vulnerabilities
  • Use read-only filesystems
docker run -d \
  --read-only \
  --cap-drop ALL \
  myapp:1.0

8) Docker Compose for sanity

version: "3.9"
services:
  web:
    image: nginx:1.25-alpine
    ports:
      - "80:80"
    restart: always

9) Common production mistakes

  • Using :latest
  • No restart policy
  • No health checks
  • No monitoring on the host
Final advice: Docker simplifies deployment, not operations. Treat containers like cattle, not pets.
